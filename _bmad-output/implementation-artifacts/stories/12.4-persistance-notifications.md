# Story 12.4 : Persistance des notifications

## Description

Les toggles de notifications (email et push) dans Settings.jsx fonctionnent visuellement mais ne sont pas persistes. Les valeurs sont en state local (`useState(true)`) et se reinitialise a `true` a chaque rechargement de page. Cette story ajoute les champs `emailNotifications` et `pushNotifications` au modele utilisateur, les persiste via l'API, et les charge au montage du composant.

## Etat actuel du code

Dans `client/src/pages/Settings.jsx` (lignes 37-39):
```javascript
const [emailNotifications, setEmailNotifications] = useState(true)
const [pushNotifications, setPushNotifications] = useState(true)
```

Les `Switch` (lignes 274-278 et 289-293) appellent `setEmailNotifications` / `setPushNotifications` mais sans aucun appel API. Au refresh, les deux reviennent a `true`.

## User Story

As a **colocataire**,
I want **que mes preferences de notifications soient sauvegardees**,
So that **mes choix persistent apres un rafraichissement de page**.

## Criteres d'Acceptation

**AC-1: Persistance email notifications**
**Given** je toggle "Notifications par e-mail" sur OFF
**When** je rafraichis la page
**Then** le toggle est toujours sur OFF

**AC-2: Persistance push notifications**
**Given** je toggle "Notifications Push" sur ON
**When** je me deconnecte et me reconnecte
**Then** le toggle est toujours sur ON

**AC-3: Chargement initial depuis l'utilisateur**
**Given** je me connecte et j'arrive sur `/settings`
**When** la page se charge
**Then** les toggles refletent les valeurs stockees dans mon profil utilisateur
**And** si les champs n'existent pas encore (migration), les valeurs par defaut sont `true`

**AC-4: Sauvegarde immediate au toggle**
**Given** je toggle une notification
**When** le toggle change d'etat
**Then** un appel API `PUT /api/users/:id` est envoye immediatement avec la nouvelle valeur
**And** aucun bouton "Sauvegarder" supplementaire n'est necessaire

**AC-5: Gestion d'erreur**
**Given** l'API echoue lors de la sauvegarde d'une preference
**When** le toggle a ete change
**Then** le toggle revient a son etat precedent
**And** un message d'erreur discret s'affiche brievement

### Task 3: Modifier l'initialisation des toggles dans Settings.jsx

**File:** `client/src/pages/Settings.jsx`

Remplacer les lignes 37-39:
```javascript
const [emailNotifications, setEmailNotifications] = useState(true)
```

```javascript

```

### Task 4: Ajouter la sauvegarde automatique au toggle

**File:** `client/src/pages/Settings.jsx`

Creer une fonction `handleToggleNotification` qui:
1. Met a jour le state local immediatement (UX optimiste)
2. Appelle `updateUser(user.id, { [field]: value })`
3. Met a jour `setUser` et `sessionStorage` avec le retour API
4. En cas d'erreur: revient a l'etat precedent

```javascript
async function handleToggleNotification(field, value) {
  const prev = field === 'emailNotifications' ? emailNotifications : pushNotifications;
  const setter = field === 'emailNotifications' ? setEmailNotifications : setPushNotifications;
  setter(value); // Optimistic update
  try {
    const updated = await updateUser(user.id, { [field]: value });
    setUser(updated);
    // Sync sessionStorage
    const saved = sessionStorage.getItem("colocapp_user");
    if (saved) {
      const data = JSON.parse(saved);
      data.user = updated;
      sessionStorage.setItem("colocapp_user", JSON.stringify(data));
    }
  } catch (err) {
    setter(prev); // Rollback on error
    setError("Erreur lors de la sauvegarde des preferences");
    setTimeout(() => setError(""), 3000);
  }
}
```


**Request Body:**
```json

```

**Response 200:**
```json

  "data": {
    "id": "user-1",
    "name": "Thomas Durand",

    "role": "admin",
    "colocationId": "coloc-1",

    "pushNotifications": true
  }
}
```

## Notes de Migration

Les fichiers JSON existants (`server/data/db/users.json`) ne contiennent pas les champs `emailNotifications` et `pushNotifications`. Deux options:
1. **Supprimer `users.json`** pour forcer la regeneration depuis `mockData.js` (perte des users ajoutes en test)
2. **Ajouter manuellement** les champs dans le JSON existant
3. **Gerer le fallback** cote client (`user?.emailNotifications ?? true`) — deja fait dans Task 3

L'option 3 est la plus sure et ne necessite aucune migration.
