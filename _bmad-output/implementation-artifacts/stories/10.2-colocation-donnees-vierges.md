# Story 10.2 : Création de colocation avec données vierges

Status: todo

## Story

As a **utilisateur**,
I want **que quand je crée une colocation, elle démarre avec des données vierges**,
So that **je ne vois pas des données pré-remplies identiques pour tout le monde**.

## Contexte du bug

Actuellement, le serveur ne gere qu'une seule colocation en memoire (variable globale `colocation` dans `server/index.js` ligne 14). Quand `POST /api/colocation` est appele, il ecrase la colocation existante via `Object.assign(colocation, { name, invitationCode: code })` (ligne 124) au lieu de creer une nouvelle entite. De plus, `POST /api/auth/register` hardcode `colocationId: 'coloc-1'` (ligne 82) et ajoute automatiquement le nouvel utilisateur a la colocation existante (ligne 86).

Le resultat : toute nouvelle "colocation" ecrase la precedente, et tous les utilisateurs voient les memes donnees mockees (taches, finances, abonnements) de `coloc-1`.

## Acceptance Criteria

1. **Given** je cree une nouvelle colocation, **When** j'accede au dashboard, **Then** les widgets affichent 0 taches, 0 EUR cagnotte, 0 articles courses, 0 abonnements.
2. **And** aucune donnee mockee n'est pre-chargee dans la nouvelle colocation.
3. **Given** un autre utilisateur cree sa propre colocation, **When** il consulte ses donnees, **Then** il voit ses propres donnees, pas les miennes.
4. **Given** une colocation est creee, **When** je regarde son `invitationCode`, **Then** il est au format `COLO-XXXX-X` et unique.
5. **And** la colocation demo `coloc-1` conserve ses donnees mockees intactes.

### Task 2 : Creer la route `POST /api/colocation` pour multi-colocation (AC: #1, #2, #4)

Actuellement la route (ligne 120-127 de `server/index.js`) ecrase la colocation unique. Elle doit creer une nouvelle entite dans le tableau.

- [ ] **2.1** Modifier la route `POST /api/colocation` dans `server/index.js`
  ```js
  // AVANT (lignes 120-127)
  app.post('/api/colocation', (req, res) => {
    const { name } = req.body || {};
    if (!name) return res.status(400).json({ error: 'Nom requis' });
    const code = `COLO-${...}`;
    Object.assign(colocation, { name, invitationCode: code });
    db.save('colocation', colocation);
    res.status(201).json({ data: colocation });
  });

  // APRES
  app.post('/api/colocation', (req, res) => {
    const { name, userId } = req.body || {};
    if (!name) return res.status(400).json({ error: 'Nom requis' });

    const code = generateInvitationCode();
    const newColoc = {
      id: genId(colocations, 'coloc'),
      name,
      invitationCode: code,
      totalFund: 0,
      createdAt: new Date().toISOString(),
      members: userId ? [userId] : [],
    };
    colocations.push(newColoc);
    db.save('colocations', colocations);

    // Mettre a jour le user pour pointer vers la nouvelle colocation
    if (userId) {
      const userIdx = users.findIndex(u => u.id === userId);
      if (userIdx !== -1) {
        users[userIdx].colocationId = newColoc.id;
        db.save('users', users);
      }
    }

    res.status(201).json({ data: enrichColocation(newColoc) });
  });
  ```

- [ ] **2.2** Ajouter la fonction `generateInvitationCode()` en helper
  ```js
  const generateInvitationCode = () => {
    const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789';
    const block = () => Array.from({ length: 4 }, () =>
      chars[Math.floor(Math.random() * chars.length)]
    ).join('');
    const suffix = chars[Math.floor(Math.random() * chars.length)];
    return `COLO-${block()}-${suffix}`;
  };
  ```

- [ ] **2.3** Verifier l'unicite du code genere
  ```js

  do {
    code = generateInvitationCode();
  } while (colocations.some(c => c.invitationCode === code));
  ```

### Task 3 : Adapter les routes GET/join colocation (AC: #3, #5)

  ```js
  });
  ```

- [ ] **3.2** Modifier `POST /api/colocation/join` pour chercher dans le tableau `colocations`
  ```js
  // AVANT (lignes 129-147) - cherche dans une seule colocation
  // APRES - chercher le code dans toutes les colocations
  app.post('/api/colocation/join', (req, res) => {
    const { code, userId } = req.body || {};
    if (!code) return res.status(400).json({ error: 'Code requis' });

    const coloc = colocations.find(
      c => c.invitationCode.toUpperCase() === code.toUpperCase()
    );
    if (!coloc) {
      return res.status(404).json({ error: "Code d'invitation invalide" });
    }

    if (userId) {
      const user = users.find(u => u.id === userId);
      if (user) {
        if (!coloc.members.includes(userId)) {
          coloc.members.push(userId);
          db.save('colocations', colocations);
        }
        user.colocationId = coloc.id;
        db.save('users', users);
      }
    }

    res.json({ data: enrichColocation(coloc) });
  });
  ```

## API Changes

### `POST /api/colocation` - Creation de colocation

**AVANT :**
```
Request:  { name: "Ma Coloc" }
Response: { data: { id: "coloc-1", name: "Ma Coloc", invitationCode: "COLO-XXXX-X", totalFund: 245.50, members: [...] } }
Effet:    Ecrase la colocation existante (Object.assign)
```

**APRES :**
```
Request:  { name: "Ma Coloc", userId: "user-5" }
Response: { data: { id: "coloc-2", name: "Ma Coloc", invitationCode: "COLO-AB3F-K", totalFund: 0, createdAt: "...", members: [{ id: "user-5", name: "...", ... }] } }
Effet:    Cree une nouvelle colocation dans le tableau colocations[], met a jour user.colocationId
```

### `GET /api/colocation` - Lecture de colocation

**AVANT :**
```
Request:  GET /api/colocation
Response: { data: { ...la seule colocation... } }
```

**APRES :**
```
Request:  GET /api/colocation?id=coloc-2
Response: { data: { id: "coloc-2", name: "Ma Coloc", totalFund: 0, members: [...] } }
Erreur:   GET /api/colocation (sans id) -> 400 { error: "colocationId requis" }
```

### `POST /api/auth/register` - Inscription

**AVANT :**
```
Response: { data: { user: { colocationId: "coloc-1" }, colocation: { id: "coloc-1", ... } } }
Effet:    Ajoute automatiquement a coloc-1
```

**APRES :**
```
Response: { data: { user: { colocationId: null }, colocation: null } }
Effet:    Aucune colocation assignee, redirect vers /onboarding
```

### `POST /api/auth/login` - Connexion

**AVANT :**
```
Response: { data: { user: {...}, colocation: { ...coloc-1 toujours... } } }
```

**APRES :**
```
Response: { data: { user: {...}, colocation: { ...colocation du user, ou null... } } }
```

## Dev Notes

### Architecture Compliance

**Stack technique :**
- Express.js (server)
- JSON file-based persistence via `server/data/db.js`
- React 18+ (client) avec Vite
- `sessionStorage` pour la session utilisateur (AuthContext)

**Patterns obligatoires :**
- camelCase pour variables et fonctions
- PascalCase pour composants React
- `fetch()` natif pour les appels API (pas axios)
- `genId(items, prefix)` pour la generation d'IDs (helper existant ligne 30 de index.js)
- `db.save(key, data)` et `db.load(key)` pour la persistance
- `enrichColocation(coloc)` pour mapper les member IDs en objets user (helper existant ligne 40-48)
- `stripPassword()` pour ne jamais exposer les mots de passe (helper existant ligne 32)

### Risques et points d'attention

1. **Migration du fichier JSON** : le fichier `server/data/db/colocation.json` existe deja avec un objet (pas un tableau). Il faut soit le migrer, soit le supprimer pour laisser `ensureFile` le recreer depuis mockData.
2. **Backward compatibility** : les utilisateurs de demo (`thomas@coloc.fr`, `lea@coloc.fr`, `marc@coloc.fr`) doivent continuer a fonctionner avec `coloc-1`.
3. **Variable globale** : passer de `let colocation = ...` (objet) a `let colocations = ...` (tableau) impacte toutes les routes qui referenceent `colocation` directement. Faire un search-and-replace methodique.
4. **Register flow** : apres inscription, si `colocation` est `null`, le routeur client doit rediriger vers `/onboarding` et non vers `/dashboard`. Verifier le routing guard dans le client.
