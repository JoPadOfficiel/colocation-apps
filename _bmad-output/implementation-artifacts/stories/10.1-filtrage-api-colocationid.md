# Story 10.1 : Filtrage API par colocationId

## Description

En tant que **colocataire**,
je veux **que les donnees affichees soient uniquement celles de ma colocation**,
afin de **ne jamais voir les taches, depenses ou membres d'une autre colocation**.

### Contexte

Actuellement, toutes les routes GET de l'API (`/api/tasks`, `/api/finances`, `/api/subscriptions`, `/api/recipes`, `/api/shopping-list`) retournent **toutes** les donnees sans aucun filtre par colocation. De plus, les routes POST hardcodent `colocationId: 'coloc-1'` au lieu d'utiliser la valeur envoyee par le client. Cela signifie que si deux colocations existent, elles voient et modifient les memes donnees.

Cette story est un **prerequis critique** pour toutes les autres corrections des epics 10 a 13.

## Acceptance Criteria

### AC1 : Filtrage des GET par colocationId

**Given** je suis connecte dans la colocation "Appart Jopad" (id: `coloc-1`)
**When** je consulte les taches, finances, abonnements, recettes ou liste de courses
**Then** seules les donnees dont `colocationId === 'coloc-1'` s'affichent
**And** les donnees des autres colocations ne sont jamais visibles

### AC2 : Query param obligatoire

**Given** un GET est envoye a `/api/tasks`, `/api/finances`, `/api/subscriptions`, `/api/recipes`, `/api/shopping-list`
**When** le query param `colocationId` est fourni (ex: `/api/tasks?colocationId=coloc-1`)
**Then** l'API retourne uniquement les enregistrements dont `colocationId` correspond

**Given** un GET est envoye sans `colocationId`
**When** l'API recoit la requete
**Then** l'API retourne une erreur 400 `{ error: "colocationId requis" }`

### AC3 : Suppression du hardcode dans les POST

**Given** je cree une tache, depense, abonnement, recette ou article
**When** le POST est envoye
**Then** le `colocationId` utilise est celui envoye dans le body par le client (depuis `useAuth().colocation.id`)
**And** le serveur ne force plus `colocationId: 'coloc-1'`

### AC4 : Le client envoie colocationId dans chaque appel

**Given** je suis connecte et `useAuth().colocation.id` retourne `coloc-1`
**When** n'importe quel fetch GET est declenche
**Then** l'URL inclut `?colocationId=coloc-1`
**And** les POST/PUT incluent `colocationId` dans le body

## Technical Tasks

### Task 1 : Middleware de validation `colocationId` sur le serveur

**Fichier** : `server/index.js`

Creer un middleware Express qui intercepte les routes de donnees et valide la presence de `colocationId`.

```javascript
// Ajouter apres la ligne 11 (app.use(express.json()))
const requireColocationId = (req, res, next) => {
  const colocationId = req.query.colocationId || req.body?.colocationId;
  if (!colocationId) {
    return res.status(400).json({ error: 'colocationId requis' });
  }
  req.colocationId = colocationId;
  next();
};
```

### Task 8 : Filtrer aussi GET /api/users par colocationId (optionnel mais recommande)

**Fichier** : `server/index.js` (ligne 102-104)

Actuellement `GET /api/users` retourne TOUS les utilisateurs. Pour l'isolation, ajouter un filtre optionnel :

```javascript
app.get('/api/users', (req, res) => {
  const colocationId = req.query.colocationId;
  if (colocationId) {
    // Trouver la colocation et filtrer par ses members
    const colocs = Array.isArray(colocation) ? colocation : [colocation];
    const coloc = colocs.find(c => c.id === colocationId);
    const memberIds = coloc?.members || [];
    const filtered = users.filter(u => memberIds.includes(u.id));
    return res.json({ data: filtered.map(stripPassword) });
  }
  res.json({ data: users.map(stripPassword) });
});
```
